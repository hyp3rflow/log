# Hyperliquid는 어떻게 동작하는가

## 1. AMM vs 주문장 DEX — 왜 주문장인가

[이전 글](/posts/amm-dex)에서 AMM DEX의 동작 원리를 다뤘다. AMM은 유동성 공급을 민주화했지만, 트레이딩 관점에서 근본적인 한계가 있다:

- **슬리피지**: 스왑 규모가 커지면 실행 가격이 크게 밀린다. CEX의 지정가 주문처럼 정확한 가격에 체결할 수 없다.
- **주문 유형 부재**: 지정가(limit), 스탑-로스(stop-loss), 이익 실현(take-profit) 같은 고급 주문을 지원하지 않는다.
- **비영구적 손실**: LP가 부담하는 구조적 비용이다.
- **무기한 선물 불가**: AMM 구조로는 레버리지 포지션, 펀딩 레이트, 청산 메커니즘을 자연스럽게 구현하기 어렵다.

CEX 수준의 트레이딩 경험을 온체인에 구현하려면 **주문장(order book)**이 필요하다. 하지만 온체인 주문장은 처리량(throughput)과 지연(latency) 문제가 있다. 이더리움 메인넷은 ~15 TPS, 블록 시간 12초다. 마켓 메이커가 주문을 수정하려면 12초를 기다려야 한다.

Hyperliquid는 이 문제를 **전용 L1 블록체인**을 만들어 해결한다.

## 2. Hyperliquid L1 아키텍처

### 왜 자체 체인인가

이더리움 L2(Arbitrum, Optimism 등)에서도 주문장 DEX를 구현할 수 있다. dYdX v3가 이 접근을 취했다가, v4에서 **자체 체인(Cosmos SDK 기반)**으로 이전했다. 이유는 동일하다:

- **처리량 제어**: L2는 시퀀서의 처리량과 가스 한도에 제약된다. 자체 체인이면 주문장 엔진에 최적화된 블록 구조와 실행 환경을 설계할 수 있다.
- **수수료 구조**: 주문 제출/취소에 가스비가 들면 마켓 메이킹이 비효율적이다. 자체 체인에서는 주문 관련 트랜잭션의 가스를 0으로 설정할 수 있다.
- **지연 최소화**: 합의 알고리즘과 네트워크 스택을 트레이딩에 맞게 최적화할 수 있다.

### HyperBFT 합의

Hyperliquid는 **HyperBFT**라는 자체 합의 알고리즘을 사용한다. HotStuff[^1]과 그 후속 연구에서 영감을 받은 BFT(Byzantine Fault Tolerant) 알고리즘이다.

HotStuff의 핵심 혁신은 **3단계 파이프라이닝**이다. 전통적 BFT(PBFT)가 O(n²) 메시지 복잡도를 가진 것과 달리, HotStuff는 리더 기반 프로토콜로 O(n) 메시지 복잡도를 달성한다:

1. **Prepare**: 리더가 블록 제안, 검증인들이 투표
2. **Pre-commit**: 2/3 이상 투표를 모아 QC(Quorum Certificate) 생성
3. **Commit**: QC를 기반으로 다음 라운드의 리더가 새 블록 제안 (파이프라이닝)

HyperBFT는 이 구조를 최적화하여 **1블록 파이널리티**를 제공한다. 블록이 커밋되면 번복되지 않는다. 이는 트레이딩에 필수적이다 — 체결된 주문이 이후 롤백되면 안 되기 때문이다.

현재 Hyperliquid는 초당 **200,000 주문**을 처리할 수 있다고 밝히고 있다[^2]. 이는 주문(order) 기준이며, 체결(trade) 기준이 아님에 주의. 주문 제출, 수정, 취소가 모두 포함된다.

## 3. HyperCore — 온체인 주문장 엔진

HyperCore는 Hyperliquid L1의 상태 실행 레이어 중 하나로, 완전히 온체인에서 동작하는 주문장 엔진이다.

### 주문장 구조

HyperCore는 **CLOB(Central Limit Order Book)**를 구현한다. 매수(bid)와 매도(ask) 주문이 가격 순서대로 정렬되고, 매칭 엔진이 체결한다.

지원하는 주문 유형:
- **지정가(Limit)**: 특정 가격에 체결
- **시장가(Market)**: 현재 최우선 가격에 즉시 체결
- **스탑-마켓/스탑-리밋**: 트리거 가격 도달 시 활성화
- **TP/SL(Take-Profit/Stop-Loss)**: 포지션에 연결된 조건부 주문

모든 주문, 취소, 체결, 청산이 온체인 트랜잭션으로 처리되며, 1블록 파이널리티를 상속받는다.

### 무기한 선물 (Perpetual Futures)

HyperCore의 핵심 상품은 **무기한 선물(perpetual futures)**이다. 만기일이 없는 선물 계약으로, 펀딩 레이트(funding rate)를 통해 선물 가격을 현물 가격에 수렴시킨다.

#### 펀딩 레이트

펀딩 레이트는 롱/숏 포지션 간에 주기적으로 정산되는 비용이다:

- 선물 가격 > 현물 가격 (프리미엄): **롱이 숏에게** 펀딩 지불 → 롱 수요 억제 → 가격 하락 유도
- 선물 가격 < 현물 가격 (디스카운트): **숏이 롱에게** 펀딩 지불 → 숏 수요 억제 → 가격 상승 유도

Hyperliquid는 1시간마다 펀딩을 정산한다. 펀딩 레이트 계산:

<Math display expr='"funding rate" = "clamp"(("mark price" - "index price") / "index price", -0.05%, 0.05%)' />

- **마크 가격(mark price)**: 주문장의 중간 가격에 기반한 공정 가격. 청산 판단에 사용.
- **인덱스 가격(index price)**: 주요 CEX의 현물 가격 중간값. 오라클이 제공.

#### 마진과 레버리지

- **교차 마진(cross margin)**: 계정의 전체 잔액을 담보로 사용. 한 포지션의 손실을 다른 포지션의 이익으로 상쇄 가능.
- **격리 마진(isolated margin)**: 특정 포지션에 할당된 담보만 사용. 청산 시 해당 포지션의 마진만 잃는다.

최대 레버리지는 자산에 따라 다르며, BTC/ETH는 최대 50x까지 지원한다.

#### 청산 엔진

포지션의 **유지 마진(maintenance margin)**이 부족하면 청산이 발생한다:

<Math display expr='"maintenance margin" = "position size" times "mark price" times "maintenance margin rate"' />

마크 가격이 청산 가격에 도달하면:

1. 청산 엔진이 포지션을 인수
2. 시장가로 포지션을 청산
3. 남은 마진에서 수수료 차감 후 반환
4. 마진이 부족하면 **보험 기금(insurance fund)**에서 보전

보험 기금은 청산 수익과 프로토콜 수수료의 일부로 충당된다. 보험 기금이 고갈되면 사회화 손실(socialized loss)이 발생할 수 있다 — 수익 포지션에서 일부를 차감한다.

## 4. 스팟 거래

HyperCore는 무기한 선물 외에 **스팟 주문장**도 지원한다. 스팟 거래는 실제 토큰의 소유권 이전이다 (선물은 차액 정산).

Hyperliquid의 스팟 마켓은 토큰 표준과 직접 통합된다. 새로운 토큰은 **HIP-1**(Hyperliquid Improvement Proposal) 표준으로 발행되며, 스팟 주문장에 즉시 등록될 수 있다. 토큰 발행과 거래소 상장이 동일한 인프라 위에서 이루어진다.

## 5. HyperEVM — 범용 스마트 컨트랙트 환경

HyperCore가 주문장 엔진이라면, **HyperEVM**은 이더리움 호환 스마트 컨트랙트 환경이다. 동일한 Hyperliquid L1 위에서 HyperCore와 함께 실행된다.

### 핵심 특징

- **EVM 호환**: Solidity/Vyper로 작성된 스마트 컨트랙트를 배포할 수 있다. 기존 이더리움 도구(Hardhat, Foundry, MetaMask)를 그대로 사용한다.
- **HyperCore 유동성 접근**: HyperEVM의 스마트 컨트랙트가 HyperCore의 주문장 유동성을 **프리컴파일(precompile)**을 통해 접근할 수 있다. DeFi 프로토콜이 HyperCore의 깊은 유동성을 활용 가능.
- **동일한 합의**: HyperBFT의 1블록 파이널리티를 HyperEVM도 동일하게 상속한다.

### HyperCore와 HyperEVM의 관계

```
┌─────────────────────────────────────┐
│          Hyperliquid L1             │
│         (HyperBFT 합의)             │
│                                     │
│  ┌──────────────┐  ┌─────────────┐  │
│  │  HyperCore   │  │  HyperEVM   │  │
│  │              │  │             │  │
│  │ - 주문장     │←→│ - 스마트     │  │
│  │ - 무기한선물  │  │   컨트랙트   │  │
│  │ - 스팟       │  │ - DeFi      │  │
│  │ - 청산엔진   │  │ - NFT 등    │  │
│  └──────────────┘  └─────────────┘  │
│         ↑ 프리컴파일 브릿지 ↑        │
└─────────────────────────────────────┘
```

이 구조의 장점은 **성능과 범용성의 분리**다. 고성능이 필요한 주문장 엔진은 HyperCore에서 네이티브로 실행하고, 범용 DeFi 로직은 HyperEVM에서 EVM으로 실행한다.

## 6. 거버넌스와 토큰 이코노미

### HYPE 토큰

HYPE는 Hyperliquid의 네이티브 토큰이다:

- **스테이킹**: 검증인 노드 운영 및 위임 스테이킹
- **수수료**: 거래 수수료 일부가 HYPE로 지불/소각
- **거버넌스**: HIP(Hyperliquid Improvement Proposal)을 통한 프로토콜 변경 투표

### 검증인 구조

Hyperliquid는 **위임 지분 증명(Delegated Proof of Stake)** 모델이다. 검증인은 HYPE를 스테이킹하고, 다른 사용자는 검증인에게 위임할 수 있다. 합의 참여의 투표 가중치는 스테이킹된 HYPE에 비례한다.

## 7. CEX vs Hyperliquid — 실제 비교

| 항목 | CEX (Binance 등) | Hyperliquid |
|---|---|---|
| 주문장 | 오프체인 | 온체인 (HyperCore) |
| 처리량 | 수백만 TPS | ~200K 주문/초 |
| 체결 지연 | &lt;1ms | ~수백ms (합의 포함) |
| 자산 보관 | 거래소 수탁 | 자기 수탁 (self-custody) |
| 투명성 | 블랙박스 | 모든 주문/체결 온체인 검증 가능 |
| 검열 저항 | 계정 동결 가능 | 검증인 합의 필요 |
| 수수료 | Maker 0.02~0.1% | Maker 0.01%, Taker 0.035% |
| KYC | 필수 | 불필요 (지갑 연결만) |

Hyperliquid의 핵심 가치 제안은 **CEX 수준의 트레이딩 경험 + 자기 수탁**이다. 사용자의 자산은 항상 본인의 지갑에 있으며, 거래소가 해킹당하거나 파산해도 자산을 잃지 않는다. FTX 파산(2022) 이후 이 가치가 더 부각되었다.

## 8. 기술적 트레이드오프

### 탈중앙화 수준

현재 Hyperliquid의 검증인 수는 수십 개 수준이다. 이더리움의 수십만 검증인과 비교하면 상당히 중앙화되어 있다. 높은 처리량을 위해 검증인 수를 제한한 것이다 — BFT 합의에서 검증인 수가 늘면 통신 오버헤드가 증가하고 지연이 커진다.

### 오라클 의존성

인덱스 가격은 외부 CEX의 현물 가격에서 가져온다. 오라클이 조작되거나 CEX가 다운되면 마크 가격이 왜곡될 수 있다. 이는 청산과 펀딩 레이트에 직접 영향을 미친다.

### 브릿지 보안

Hyperliquid에 자금을 입금하려면 이더리움 등 다른 체인에서 **브릿지**를 통해 전송해야 한다. 브릿지는 크로스체인 보안의 가장 취약한 지점이다. 2022년 Wormhole ($320M), Ronin ($625M) 등 대규모 브릿지 해킹이 있었다.

### 코드 비공개

Hyperliquid L1의 소스코드는 현재 공개되어 있지 않다. 검증인 소프트웨어의 동작을 독립적으로 감사할 수 없다는 의미다. 이는 "온체인"이지만 "오픈소스"는 아닌 독특한 위치다.

## 9. 정리

| | AMM DEX (Uniswap) | 주문장 DEX (Hyperliquid) |
|---|---|---|
| 가격 결정 | 수학 공식 (x·y=k) | 주문 매칭 |
| 유동성 공급 | 누구나 LP 가능 | 마켓 메이커 필요 |
| 슬리피지 | 구조적 (풀 크기 대비) | 주문장 깊이에 따라 |
| 주문 유형 | 스왑만 | Limit, Stop, TP/SL 등 |
| 레버리지 | 제한적 | 최대 50x |
| 실행 환경 | 이더리움/L2 | 전용 L1 |
| 트레이드오프 | IL, MEV | 중앙화, 오라클 의존 |

AMM과 주문장은 서로 다른 문제를 해결한다. AMM은 "누구나 유동성을 공급하고 거래할 수 있는" 접근성에 초점을 맞추고, 주문장은 "CEX 수준의 트레이딩 기능을 온체인에서" 제공하는 데 초점을 맞춘다. Hyperliquid는 후자를 위해 전용 L1까지 만든 극단적 케이스다.
