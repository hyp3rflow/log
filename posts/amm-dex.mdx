## 온체인 주문장의 한계

중앙화 거래소(CEX)는 주문장(order book)을 사용한다. 매수자와 매도자가 각각 가격과 수량을 제출하고, 매칭 엔진이 체결한다. 이 모델은 오프체인에서는 효율적이지만, 블록체인 위에 올리면 구조적 문제가 발생한다.

**가스비**: 주문 제출, 수정, 취소 모두 온체인 트랜잭션이다. 시장 조성자(market maker)가 가격 변동에 맞춰 주문을 갱신하려면 초당 수십 건의 트랜잭션이 필요한데, 이더리움 메인넷 기준 한 건당 수만 원의 가스비가 든다.

**지연(latency)**: 이더리움의 블록 생성 시간은 약 12초다. 주문장 기반 트레이딩에서 12초의 지연은 차익거래자에게 무제한 기회를 제공한다.

**유동성 파편화**: 온체인 주문장은 체인별, 컨트랙트별로 유동성이 분산된다. 충분한 깊이(depth)를 확보하기 어렵다.

이 문제를 해결하기 위해 등장한 것이 AMM(Automated Market Maker)이다. AMM은 주문장을 수학 공식으로 대체한다. 매수/매도 주문 없이, 유동성 풀에 예치된 토큰 쌍과 불변식(invariant)만으로 가격을 결정한다. Bancor(2017)가 초기 개념을 제시했고, Uniswap(2018)이 이를 단순하고 강력한 형태로 구현했다.

## Constant Product Market Maker

Uniswap v1/v2의 핵심은 Constant Product Formula다:

<Math display expr='x dot y = k' />

여기서 <Math expr='x' />는 토큰 A의 리저브, <Math expr='y' />는 토큰 B의 리저브, <Math expr='k' />는 상수다. 스왑이 발생해도 <Math expr='k' />는 변하지 않는다(수수료 제외).

### 가격 결정

토큰 A를 <Math expr='Delta x' />만큼 풀에 넣고 토큰 B를 <Math expr='Delta y' />만큼 받는 스왑을 생각한다. 수수료를 무시하면:

<Math display expr='(x + Delta x)(y - Delta y) = k = x dot y' />

이를 <Math expr='Delta y' />에 대해 풀면:

<Math display expr='Delta y = y - (x dot y) / (x + Delta x) = (y dot Delta x) / (x + Delta x)' />

스팟 가격(marginal price)은 <Math expr='Delta x -> 0' />일 때의 비율이다:

<Math display expr='"price" = (d y) / (d x) = y / x' />

리저브 비율이 곧 가격이다. 누군가 토큰 A를 대량으로 스왑하면 <Math expr='x' />가 증가하고 <Math expr='y' />가 감소하여, 토큰 A의 가격(B 기준)이 하락한다. 이것이 자동 가격 발견 메커니즘이다.

### 슬리피지

스왑 금액이 클수록 실행 가격은 스팟 가격에서 멀어진다. 슬리피지는 다음과 같이 정의된다:

<Math display expr='"slippage" = ("spot price" - "exec price") / "spot price"' />

<Math expr='Delta x' />가 <Math expr='x' />에 비해 클수록 슬리피지가 커진다. 구체적으로:

<Math display expr='"exec price" = (Delta y) / (Delta x) = y / (x + Delta x)' />

<Math display expr='"slippage" = 1 - x / (x + Delta x) = (Delta x) / (x + Delta x)' />

리저브가 1,000 ETH인 풀에서 100 ETH를 스왑하면 슬리피지는 약 9.1%다.

### 수수료 구조

Uniswap v2는 모든 스왑에 0.3% 수수료를 부과한다. 수수료는 입력 토큰에서 차감된다:

<Math display expr='Delta x_"eff" = Delta x dot 0.997' />

<Math display expr='(x + Delta x_"eff")(y - Delta y) = k' />

수수료는 풀의 리저브에 누적되어 <Math expr='k' />가 점진적으로 증가한다. 이 증가분이 LP의 수익이다.

## 유동성 공급과 비영구적 손실

### LP 토큰

유동성 공급자(LP)는 토큰 쌍을 현재 가격 비율에 맞춰 예치한다. 그 대가로 LP 토큰을 발급받는다. Uniswap v2에서 LP 토큰 발행량은 다음과 같다:

최초 공급 시:

<Math display expr='s_"minted" = sqrt(x dot y)' />

이후 추가 공급 시:

<Math display expr='s_"minted" = min((Delta x) / x, (Delta y) / y) dot S' />

여기서 <Math expr='S' />는 현재 총 LP 토큰 공급량이다. 인출 시에는 LP 토큰을 소각하고, 풀에서 비례적으로 토큰을 돌려받는다.

### 비영구적 손실(Impermanent Loss)

LP가 토큰을 예치한 후, 외부 시장에서 가격이 변동하면 차익거래자가 풀의 가격을 맞추기 위해 스왑한다. 이 과정에서 LP는 "보유했더라면 얻었을 가치"보다 적은 가치를 갖게 된다. 이를 비영구적 손실이라 한다.

가격 변동 비율을 <Math expr='r = p_1 / p_0' />로 정의한다. 초기 가격 <Math expr='p_0' />에서 예치한 LP의 자산 가치 대비, 가격이 <Math expr='p_1' />으로 변한 후의 가치 비율은:

<Math display expr='"IL"(r) = (2 sqrt(r)) / (1 + r) - 1' />

유도 과정은 다음과 같다. 초기 상태에서 <Math expr='x_0, y_0' />를 예치했고 <Math expr='p_0 = y_0 / x_0' />이다. 가격이 <Math expr='r' />배 변하면:

<Math display expr='x_1 = x_0 / sqrt(r), quad y_1 = y_0 dot sqrt(r)' />

LP의 풀 내 자산 가치:

<Math display expr='V_"pool" = x_1 dot p_1 + y_1 = x_0 dot sqrt(r) dot p_0 dot r + y_0 dot sqrt(r)' />

이를 정리하면:

<Math display expr='V_"pool" = 2 y_0 sqrt(r)' />

단순 보유(HODL) 시 가치:

<Math display expr='V_"hold" = x_0 dot p_1 + y_0 = y_0(1 + r)' />

따라서:

<Math display expr='"IL"(r) = V_"pool" / V_"hold" - 1 = (2 sqrt(r)) / (1 + r) - 1' />

| 가격 변동(<Math expr='r' />) | 비영구적 손실 |
|---|---|
| 1.25 (25% 상승) | −0.6% |
| 1.50 (50% 상승) | −2.0% |
| 2.00 (2배) | −5.7% |
| 3.00 (3배) | −13.4% |
| 5.00 (5배) | −25.5% |

가격이 2배가 되면 약 5.7%의 손실이 발생한다. "비영구적"이라 부르는 이유는 가격이 원래로 돌아오면 손실이 사라지기 때문이다. 그러나 현실에서 가격이 정확히 복귀하는 경우는 드물다.

## Uniswap v3 — Concentrated Liquidity

Uniswap v2에서 유동성은 가격 0부터 무한대까지 균일하게 분포한다. 대부분의 거래가 현재 가격 근처에서 발생함에도 자본의 대부분이 사용되지 않는다. Uniswap v3(2021)는 이 문제를 해결하기 위해 <strong>Concentrated Liquidity</strong>를 도입했다.

### 틱(Tick) 시스템

가격 공간을 이산적인 틱으로 나눈다. 각 틱 <Math expr='i' />는 가격 <Math expr='p(i) = 1.0001^i' />에 대응한다. LP는 하한 틱 <Math expr='i_l' />과 상한 틱 <Math expr='i_u' /> 사이에만 유동성을 집중시킬 수 있다.

Uniswap v3에서 유동성 <Math expr='L' />은 다음과 같이 정의된다:

<Math display expr='L = sqrt(x dot y)' />

현재 가격 <Math expr='P' />가 LP의 구간 <Math expr='[p_a, p_b]' /> 안에 있을 때, 토큰 보유량은:

<Math display expr='x = L dot (1 / sqrt(P) - 1 / sqrt(p_b))' />

<Math display expr='y = L dot (sqrt(P) - sqrt(p_a))' />

### 자본 효율성

가격 구간을 좁게 설정할수록 같은 유동성 <Math expr='L' />을 더 적은 자본으로 달성할 수 있다. ETH/USDC 풀에서 가격 범위를 현재가의 ±0.1%로 설정하면, v2 대비 최대 4,000배의 자본 효율성을 달성할 수 있다(Uniswap v3 Whitepaper).

단, 가격이 LP의 설정 구간을 벗어나면 유동성이 비활성화된다. 한쪽 토큰만 남게 되며 수수료를 벌지 못한다. 이는 LP의 적극적 관리(active management)를 요구한다.

### 수수료 티어

v3는 0.01%, 0.05%, 0.30%, 1.00%의 네 가지 수수료 티어를 제공한다. 스테이블코인 페어는 낮은 수수료를, 변동성 높은 페어는 높은 수수료를 선택하는 것이 일반적이다. 틱 간격은 수수료 티어에 연동된다 — 0.01% 풀은 1틱, 0.05%는 10틱, 0.30%는 60틱, 1.00%는 200틱 간격이다.

## Uniswap v4 — Singleton과 Hooks

2024년에 출시된 Uniswap v4는 아키텍처를 근본적으로 변경했다.

### 싱글톤 아키텍처

v2/v3에서는 각 풀이 독립적인 컨트랙트였다. v4는 모든 풀을 하나의 `PoolManager` 컨트랙트에 통합한다. 이를 싱글톤 패턴이라 한다.

멀티홉 스왑(A→B→C)에서 v3는 각 풀 컨트랙트 간 실제 토큰 전송이 필요했다. v4는 <strong>flash accounting</strong>을 사용한다. 스왑 과정에서 실제 토큰은 이동하지 않고, 내부 잔액(delta)만 추적한다. 트랜잭션 종료 시 최종 순잔액(net balance)만 정산한다. 이를 통해 가스비를 대폭 절감한다.

### Hooks

v4의 가장 핵심적인 혁신은 hooks 시스템이다. 풀 생성 시 hook 컨트랙트를 지정할 수 있으며, 다음 시점에 커스텀 로직을 삽입할 수 있다:

- `beforeInitialize` / `afterInitialize` — 풀 생성 시
- `beforeAddLiquidity` / `afterAddLiquidity` — 유동성 추가 시
- `beforeRemoveLiquidity` / `afterRemoveLiquidity` — 유동성 제거 시
- `beforeSwap` / `afterSwap` — 스왑 시
- `beforeDonate` / `afterDonate` — 수수료 기부 시

이를 통해 동적 수수료, TWAP 오라클, 리밋 오더, 온체인 옵션 등 기존에는 별도 프로토콜이 필요했던 기능을 풀 레벨에서 구현할 수 있다. hook 주소는 특정 비트 패턴을 따라야 하며, CREATE2로 배포 주소를 결정한다.

## Uniswap 포크 — PancakeSwap과 그 외

Uniswap v2 코드는 GPL-2.0으로 공개되어 있고, 수많은 포크가 존재한다. 대표적인 것이 PancakeSwap(BSC 체인)이다.

| 항목 | Uniswap v2 | PancakeSwap v2 |
|---|---|---|
| 체인 | Ethereum | BNB Smart Chain |
| 스왑 수수료 | 0.30% | 0.25% |
| LP 수수료 분배 | 전액 LP | 0.17% LP, 0.03% Treasury, 0.05% Buyback |
| 추가 기능 | — | Lottery, IFO, Prediction |
| AMM 공식 | x·y=k | x·y=k |

핵심 AMM 로직은 동일하다. `Router`, `Factory`, `Pair` 패턴이 거의 그대로 사용된다. 차이는 주로 수수료 분배, 거버넌스 토큰 유틸리티, 부가 기능(게임파이)에 있다. SushiSwap, Trader Joe, QuickSwap 등도 같은 구조다.

## MEV와 프론트러닝

DEX 트레이딩의 가장 심각한 구조적 문제는 MEV(Maximal Extractable Value)다.

### 샌드위치 공격

1. 공격자가 멤풀(mempool)에서 대량 스왑 트랜잭션을 감지한다.
2. 공격자는 해당 트랜잭션 **앞에** 같은 방향의 스왑을 삽입한다(프론트런). 이로써 가격을 올린다.
3. 피해자의 트랜잭션이 더 높은 가격에 체결된다.
4. 공격자는 피해자 트랜잭션 **뒤에** 반대 방향 스왑을 삽입한다(백런). 차익을 실현한다.

피해자가 설정한 슬리피지 허용치(slippage tolerance)가 공격자의 수익 상한이 된다. 슬리피지 허용치를 1%로 설정하면, 공격자는 최대 1%의 가격 변동 내에서 이익을 추출할 수 있다. 허용치를 너무 낮게 설정하면 트랜잭션이 revert될 확률이 높아진다.

### Flashbots

Flashbots는 MEV 문제에 대한 대응으로 등장했다(Daian et al., "Flash Boys 2.0", 2019에서 문제를 정의). Flashbots는 MEV-Boost를 통해 블록 빌더(builder)와 검증자(proposer) 간의 시장을 형성하고, MEV를 투명하게 추출·분배한다. 트랜잭션을 공개 멤풀 대신 프라이빗 채널로 전송하여 샌드위치 공격을 방지하는 Flashbots Protect 같은 도구도 제공한다.

## 스마트 컨트랙트 구조 — Uniswap v2

Uniswap v2의 컨트랙트 아키텍처는 Factory와 Pair로 구성된다.

### Factory 패턴

`UniswapV2Factory`는 토큰 페어별로 `UniswapV2Pair` 컨트랙트를 배포한다. CREATE2를 사용하여 배포 주소가 결정적(deterministic)이다:

```solidity filename="UniswapV2Factory.sol"
function createPair(address tokenA, address tokenB) external returns (address pair) {
    require(tokenA != tokenB, 'IDENTICAL_ADDRESSES');
    (address token0, address token1) = tokenA < tokenB
        ? (tokenA, tokenB) : (tokenB, tokenA);
    require(token0 != address(0), 'ZERO_ADDRESS');
    require(getPair[token0][token1] == address(0), 'PAIR_EXISTS');

    bytes memory bytecode = type(UniswapV2Pair).creationCode;
    bytes32 salt = keccak256(abi.encodePacked(token0, token1));
    assembly {
        pair := create2(0, add(bytecode, 32), mload(bytecode), salt)
    }

    IUniswapV2Pair(pair).initialize(token0, token1);
    getPair[token0][token1] = pair;
    getPair[token1][token0] = pair;
    allPairs.push(pair);
}
```

CREATE2의 이점은 페어 주소를 오프체인에서 계산할 수 있다는 것이다:

<Math display expr='"addr" = "keccak256"(0"xff", "factory", "salt", "initCodeHash")' />

라우터 컨트랙트가 Factory를 호출하지 않고도 페어 주소를 직접 계산할 수 있어 가스를 절약한다.

### swap() 함수

`UniswapV2Pair.swap()`이 실제 토큰 교환을 수행한다:

```solidity filename="UniswapV2Pair.sol"
function swap(uint amount0Out, uint amount1Out, address to, bytes calldata data) external lock {
    require(amount0Out > 0 || amount1Out > 0, 'INSUFFICIENT_OUTPUT_AMOUNT');
    (uint112 _reserve0, uint112 _reserve1,) = getReserves();
    require(amount0Out < _reserve0 && amount1Out < _reserve1, 'INSUFFICIENT_LIQUIDITY');

    uint balance0;
    uint balance1;
    {
        address _token0 = token0;
        address _token1 = token1;
        require(to != _token0 && to != _token1, 'INVALID_TO');
        if (amount0Out > 0) _safeTransfer(_token0, to, amount0Out);
        if (amount1Out > 0) _safeTransfer(_token1, to, amount1Out);
        if (data.length > 0) IUniswapV2Callee(to).uniswapV2Call(msg.sender, amount0Out, amount1Out, data);
        balance0 = IERC20(_token0).balanceOf(address(this));
        balance1 = IERC20(_token1).balanceOf(address(this));
    }

    uint amount0In = balance0 > _reserve0 - amount0Out ? balance0 - (_reserve0 - amount0Out) : 0;
    uint amount1In = balance1 > _reserve1 - amount1Out ? balance1 - (_reserve1 - amount1Out) : 0;
    require(amount0In > 0 || amount1In > 0, 'INSUFFICIENT_INPUT_AMOUNT');
    {
        uint balance0Adjusted = balance0.mul(1000).sub(amount0In.mul(3));
        uint balance1Adjusted = balance1.mul(1000).sub(amount1In.mul(3));
        require(
            balance0Adjusted.mul(balance1Adjusted) >= uint(_reserve0).mul(_reserve1).mul(1000**2),
            'K'
        );
    }

    _update(balance0, balance1, _reserve0, _reserve1);
    emit Swap(msg.sender, amount0In, amount1In, amount0Out, amount1Out, to);
}
```

핵심 패턴은 "optimistic transfer"다. 먼저 토큰을 보내고(`_safeTransfer`), 이후에 불변식(<Math expr='k' />)을 검증한다. `data.length > 0`이면 콜백을 호출하는데, 이것이 flash swap(플래시 론의 일종)을 가능하게 한다. 입력 토큰을 아직 받지 않은 상태에서 출력 토큰을 먼저 받고, 콜백 내에서 입력 토큰을 갚으면 된다.

수수료 검증은 `balance0Adjusted.mul(balance1Adjusted) >= reserve0 * reserve1 * 1000000` 으로 이루어진다. 입력에서 0.3%(3/1000)를 차감한 후의 곱이 기존 <Math expr='k' />를 유지해야 한다.

## 정리

AMM DEX는 주문장 없이 수학적 불변식만으로 토큰 교환을 가능하게 한다. Uniswap v2의 <Math expr='x dot y = k' />는 가장 단순하면서도 강력한 구현이었고, v3는 자본 효율성을, v4는 확장성과 커스터마이징을 해결했다. 비영구적 손실과 MEV는 여전히 AMM의 근본적 한계로 남아 있으며, 이를 완화하기 위한 연구(Uniswap v4 hooks, MEV-aware 프로토콜 등)가 진행 중이다.
