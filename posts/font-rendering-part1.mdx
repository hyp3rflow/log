# Font Rendering의 원리 — Part 1: 벡터에서 픽셀로

## 1. 문제 정의

화면에 텍스트가 표시되려면 **수학적 곡선으로 정의된 글리프 아웃라인**을 **이산적인 픽셀 격자**로 변환해야 한다. 이 과정을 래스터화(rasterization)라 한다.

왜 이것이 어려운가:

- **해상도 의존성**: 같은 글리프가 12px과 96px에서 전혀 다른 모양이 되어야 한다. 12px에서는 곡선의 미세한 디테일이 1픽셀 이하로 떨어진다.
- **정보 손실**: 연속적인 곡선을 이산 격자에 매핑하면 필연적으로 앨리어싱이 발생한다. 나이퀴스트 정리에 따르면, 픽셀 격자의 샘플링 주파수보다 높은 주파수 성분은 올바르게 표현할 수 없다.
- **지각적 품질**: 인간의 시각 시스템은 텍스트에 극도로 민감하다. 1픽셀의 위치 차이가 글자의 "무게감"이나 가독성을 바꾼다.

폰트 렌더링 파이프라인의 전체 흐름은 다음과 같다:

1. **문자 → 글리프 매핑** (cmap 테이블)
2. **텍스트 셰이핑** (GSUB/GPOS, 리거처, 커닝)
3. **스케일링 + 힌팅** (그리드 피팅)
4. **래스터화** (벡터 → 비트맵)
5. **안티앨리어싱/서브픽셀 렌더링**
6. **합성** (배경 위에 블렌딩)

이 글에서는 2~5단계를 집중적으로 다룬다. 텍스트 셰이핑과 GPU 렌더링은 [Part 2](/posts/font-rendering-part2)에서 다룬다.

## 2. 폰트 포맷과 윤곽선 데이터

### 2.1 TrueType: 2차 베지어 곡선

TrueType은 Apple이 1991년에 개발한 포맷이다. 글리프 윤곽선은 **2차 베지어 곡선(quadratic Bézier curve)**으로 정의된다.

2차 베지어 곡선의 수학적 정의:

<Math display expr='B(t) = (1-t)^2 P_0 + 2(1-t)t P_1 + t^2 P_2, quad t in [0,1]' />

여기서 <Math expr='P_0' />, <Math expr='P_2' />는 온커브(on-curve) 포인트, <Math expr='P_1' />은 오프커브(off-curve) 컨트롤 포인트다.

<BezierCurveEditor />

TrueType 폰트 파일의 핵심 테이블:

| 테이블 | 역할 |
|--------|------|
| `cmap` | 유니코드 코드포인트 → 글리프 인덱스 매핑 |
| `glyf` | 글리프 윤곽선 데이터 (포인트 좌표, 온/오프커브 플래그) |
| `loca` | `glyf` 테이블 내 각 글리프의 오프셋 |
| `head` | 전역 메타데이터 (units per em, 바운딩 박스) |
| `hmtx` | 수평 메트릭 (어드밴스 폭, 왼쪽 사이드 베어링) |
| `maxp` | 최대 프로파일 (글리프 수, 최대 포인트 수) |
| `fpgm` | 폰트 프로그램 (힌팅 바이트코드) |
| `cvt ` | 컨트롤 밸류 테이블 (힌팅 파라미터) |

`glyf` 테이블의 단일 글리프 구조는 다음과 같다:

```c filename="glyf_header.h"
typedef struct {
    int16_t  numberOfContours;  // 음수이면 복합 글리프
    int16_t  xMin, yMin, xMax, yMax;  // 바운딩 박스
    // 이후: endPtsOfContours[], instructionLength, instructions[], flags[], xCoordinates[], yCoordinates[]
} GlyfHeader;
```

각 포인트에는 플래그 바이트가 있고, bit 0이 1이면 온커브 포인트, 0이면 오프커브 포인트다. TrueType의 특징적인 최적화로, 연속된 두 오프커브 포인트 사이에는 **암묵적 온커브 포인트**가 두 포인트의 중점에 자동 삽입된다. 이를 통해 저장 공간을 절약한다.

> **참고**: Apple TrueType Reference Manual, "The `glyf` table" 참조. Microsoft OpenType Specification §5.3.4.

### 2.2 OpenType/CFF: 3차 베지어 곡선

Adobe의 PostScript 기반 CFF(Compact Font Format) 아웃라인은 **3차 베지어 곡선(cubic Bézier curve)**을 사용한다:

<Math display expr='B(t) = (1-t)^3 P_0 + 3(1-t)^2 t P_1 + 3(1-t)t^2 P_2 + t^3 P_3, quad t in [0,1]' />

3차 곡선은 두 개의 컨트롤 포인트를 사용하므로 2차 곡선보다 표현력이 높다. 동일한 형태를 더 적은 수의 세그먼트로 표현할 수 있다. 반대로, 2차 곡선은 래스터화가 약간 더 빠르다.

CFF 데이터는 PostScript Type 2 CharString으로 인코딩된다. 주요 오퍼레이터:

| 오퍼레이터 | 설명 |
|------------|------|
| `rmoveto` | 새 서브패스 시작 |
| `rlineto` | 직선 세그먼트 |
| `rrcurveto` | 3차 베지어 곡선 (dx1, dy1, dx2, dy2, dx3, dy3) |
| `hintmask` | 스템 힌트 활성화 마스크 |
| `hstem`, `vstem` | 수평/수직 스템 힌트 선언 |

> **참고**: Microsoft OpenType Specification, "CFF — Compact Font Format Table", Adobe Technical Note #5177 "The Type 2 Charstring Format".

### 2.3 비교

| | TrueType | CFF/OpenType |
|---|---|---|
| 곡선 차수 | 2차 (quadratic) | 3차 (cubic) |
| 컨트롤 포인트/세그먼트 | 1개 | 2개 |
| 힌팅 방식 | 바이트코드 VM | 스템/블루존 선언 |
| 글리프 데이터 | `glyf` 테이블 (바이너리) | `CFF ` 테이블 (PostScript CharString) |
| 파일 크기 | 비슷 (암묵적 온커브로 절약) | 비슷 (적은 세그먼트 수) |
| 래스터화 비용 | 약간 낮음 | 약간 높음 |

3차 베지어를 2차로 변환하는 것은 항상 가능하지만 정확한 변환이 아닌 근사(approximation)가 필요하다. 반대로 2차→3차 변환은 정확하다 (<Math expr='P_1' />을 두 컨트롤 포인트로 분할하면 된다).

## 3. 래스터화

### 3.1 핵심 문제: 어떤 픽셀이 "안쪽"인가

벡터 아웃라인은 닫힌 곡선(contour)의 집합이다. 래스터화란 각 픽셀이 이 윤곽선의 안쪽인지 바깥인지를 판정하는 것이다.

**와인딩 넘버 규칙(Winding Number Rule)**은 이 판정에 사용되는 두 가지 표준 방법이다:

**Non-zero rule**: 한 점에서 무한 원점으로 반직선(ray)을 쏴서, 윤곽선이 반시계 방향으로 교차하면 +1, 시계 방향이면 -1을 더한다. 합이 0이 아니면 내부.

**Even-odd rule**: 교차 횟수만 세서, 홀수면 내부, 짝수면 외부.

TrueType은 non-zero 규칙을 사용하고, PostScript/CFF도 non-zero가 기본이다. SVG는 두 규칙 모두 지원한다.

### 3.2 스캔라인 래스터화

실제 래스터라이저는 점 단위 ray-casting 대신 **스캔라인 알고리즘**을 사용한다:

1. 글리프의 모든 곡선 세그먼트를 수평 스캔라인과의 교차점으로 분해
2. 각 스캔라인에서 교차점을 x좌표 순으로 정렬
3. 와인딩 규칙에 따라 내부 구간을 결정
4. 해당 구간의 픽셀을 채움

### 3.3 커버리지 계산과 안티앨리어싱

단순 이진(on/off) 래스터화는 계단 현상(jaggies)을 만든다. 이를 해결하기 위해 **커버리지(coverage)** 기반 래스터화를 사용한다:

각 픽셀을 N×N 서브픽셀로 분할하고, 윤곽선 내부에 있는 서브픽셀의 비율을 커버리지 값으로 사용한다. 이 값이 해당 픽셀의 알파(투명도) 값이 된다.

FreeType의 래스터라이저(`smooth` 렌더러, `src/smooth/ftgrays.c`)는 이보다 정교한 방식을 사용한다. Raph Levien이 분석한 바에 따르면, FreeType은 **정확한 해석적 커버리지 계산(exact analytical area computation)**을 수행한다:

1. 각 스캔라인의 1픽셀 높이 밴드에서, 곡선 세그먼트가 해당 픽셀과 교차하는 영역의 면적을 정확하게 계산
2. 베지어 곡선을 직선 세그먼트로 플래트닝(flattening)한 후 사다리꼴 면적 공식을 적용
3. 256단계 그레이스케일 커버리지 값 출력

```c filename="ftgrays.c (simplified)"
/* FreeType smooth rasterizer의 핵심 루프 (단순화) */
static void
gray_render_scanline(RAS_ARG_ TCoord ey, TPos x1, TPos y1, TPos x2, TPos y2)
{
    /* 한 스캔라인 내에서 직선 세그먼트 (x1,y1)→(x2,y2)가 
       각 픽셀 컬럼에 기여하는 커버리지를 계산 */
    TCoord  ex1, ex2, fx1, fx2;
    TArea   area, delta;
    
    ex1 = TRUNC(x1);  /* 시작점이 속한 픽셀 */
    ex2 = TRUNC(x2);  /* 끝점이 속한 픽셀 */
    fx1 = FRACT(x1);  /* 픽셀 내 소수점 위치 */
    fx2 = FRACT(x2);

    delta = y2 - y1;  /* 높이 변화량 */
    
    if (ex1 == ex2) {
        /* 한 픽셀 내에서 시작하고 끝나는 경우 */
        area = (fx1 + fx2) * delta;
        ras.area += area;
        ras.cover += delta;
    } else {
        /* 여러 픽셀에 걸치는 경우: 각 픽셀별 면적 분배 */
        /* ... */
    }
}
```

> **참고**: FreeType 소스코드 `src/smooth/ftgrays.c`, Raph Levien, "Inside the fastest font renderer in the world" (2016).

### 3.4 곡선 플래트닝

베지어 곡선을 직선 세그먼트로 분해(flattening)하는 것은 래스터화의 전처리 단계다. FreeType은 de Casteljau 알고리즘을 사용하여 재귀적으로 곡선을 이등분하고, 직선과의 오차가 임계값(보통 1/64 픽셀) 이하가 되면 중단한다.

2차 베지어의 de Casteljau 분할:

<Math display expr='P_(0 1) = (1-t) P_0 + t P_1' />
<Math display expr='P_(1 2) = (1-t) P_1 + t P_2' />
<Math display expr='P_(0 1 2) = (1-t) P_(0 1) + t P_(1 2)' />

<Math expr='t = 0.5' />에서 분할하면 원래 곡선을 두 개의 2차 베지어로 정확히 분리할 수 있다.

## 4. 힌팅

### 4.1 왜 힌팅이 필요한가

12px 크기의 글꼴에서 스템(stem, 글자의 세로 획) 두께가 1.3픽셀이라 하자. 래스터라이저는 이를 1픽셀 또는 2픽셀로 렌더링해야 한다. 아무 조정 없이 반올림하면:

- 같은 폰트의 'l'과 'i'의 스템 두께가 달라질 수 있음
- 'H'의 가로 획이 세로 획보다 가늘어 보일 수 있음
- 세리프의 대칭이 깨질 수 있음

**힌팅(hinting)**은 글리프의 주요 특징점을 픽셀 격자에 정렬(grid-fitting)하여 일관성을 유지하는 기술이다.

### 4.2 TrueType 힌팅: 바이트코드 VM

TrueType 힌팅은 전용 **바이트코드 가상 머신**으로 구현된다. 폰트 디자이너가 각 글리프에 인스트럭션(instruction)을 작성하여, 특정 크기에서 포인트의 위치를 조정한다.

주요 인스트럭션 카테고리:

| 인스트럭션 | 설명 |
|------------|------|
| `MDAP[R]` | Move Direct Absolute Point — 포인트를 그리드에 반올림 |
| `MDRP[>]` | Move Direct Relative Point — 참조점 기준 상대 이동 |
| `MIRP[]` | Move Indirect Relative Point — CVT 값 참조 이동 |
| `IUP[x/y]` | Interpolate Untouched Points — 조정되지 않은 포인트를 보간 |
| `SRP0/1/2` | Set Reference Point — 참조 포인트 설정 |
| `SVTCA[x/y]` | Set freedom/projection Vector To Coordinate Axis |

이 VM은 스택 기반이며, 26개의 그래픽스 상태 변수(freedom vector, projection vector, round state 등)를 가진다. CVT(Control Value Table)에 저장된 값을 참조하여 스템 두께, x-height 등의 일관성을 보장한다.

```text filename="TrueType hinting pseudocode"
/* 'H'의 왼쪽 스템 힌팅 예시 */
SVTCA[Y]          /* Y축으로 프로젝션/프리덤 벡터 설정 */
SRP0[], pt_baseline
MDAP[R], pt_baseline   /* 베이스라인 포인트를 그리드에 정렬 */
MIRP[01100], pt_top, cvt_cap_height  /* CVT의 캡 높이 참조하여 상단 정렬 */
SVTCA[X]          /* X축으로 전환 */
SRP0[], pt_left_outer
MDAP[R], pt_left_outer /* 왼쪽 외곽을 그리드에 정렬 */
MIRP[10100], pt_left_inner, cvt_stem_width /* 스템 폭을 CVT 값으로 제어 */
IUP[X]            /* 나머지 포인트 보간 */
IUP[Y]
```

> **참고**: Apple TrueType Reference Manual, "The TrueType Instruction Set"; Microsoft OpenType Specification §6.

### 4.3 PostScript/CFF 힌팅: 스템과 블루존

CFF 힌팅은 선언적이다. 바이트코드를 실행하는 대신, 래스터라이저에게 **힌트를 제공**하고 해석은 래스터라이저에 맡긴다.

두 가지 핵심 메커니즘:

**스템 힌트(stem hints)**: `hstem`과 `vstem` 오퍼레이터로 수평/수직 스템의 위치와 두께를 선언한다. 래스터라이저는 이를 참고하여 스템 경계를 픽셀 에지에 정렬한다.

```text filename="CFF stem hint example"
/* 'H' 글리프의 스템 힌트 */
80 100 hstem        /* y=80에서 높이 100인 수평 스템 (가로 획) */
0 700 vstem         /* 전체 높이에 걸친 영역 선언 */
100 80 vstem        /* x=100에서 폭 80인 수직 스템 (왼쪽 세로 획) */
420 80 vstem        /* x=420에서 폭 80인 수직 스템 (오른쪽 세로 획) */
```

**블루존(blue zones)**: 폰트 전역 정렬 구역이다. Private DICT의 `BlueValues` 배열로 정의되며, 베이스라인, x-height, 캡 높이, 어센더, 디센더 등의 정렬 위치를 지정한다. 같은 블루존에 속하는 글리프 특징은 동일한 픽셀 행에 정렬된다.

### 4.4 FreeType 자동 힌터

FreeType의 `autofit` 모듈(`src/autofit/`)은 힌트가 없거나 부실한 폰트에 자동으로 힌팅을 적용한다:

1. **스크립트 감지**: 글리프가 속한 스크립트(Latin, CJK, Indic 등)를 판별
2. **블루존 추출**: 기준 글리프(`o`, `O`, `x` 등)의 상단/하단에서 정렬 구역을 자동 추출
3. **세그먼트/에지 감지**: 윤곽선에서 수직/수평 스템을 자동 검출
4. **그리드 피팅**: 검출된 에지를 픽셀 경계에 정렬

> **참고**: FreeType 문서 "The auto-hinter"; 소스코드 `src/autofit/aflatin.c`.

## 5. 안티앨리어싱

### 5.1 그레이스케일 안티앨리어싱

가장 기본적인 방식이다. 각 픽셀의 커버리지 값(0.0~1.0)을 알파 채널에 매핑한다. 전경색이 검은색이고 배경이 흰색일 때, 커버리지 0.7인 픽셀은 밝기 76 (= 255 × (1 - 0.7))이 된다.

FreeType에서는 `FT_RENDER_MODE_NORMAL`로 렌더링하면 256단계 그레이스케일 비트맵을 얻는다.

### 5.2 서브픽셀 렌더링

LCD 패널의 각 픽셀은 R, G, B 세 개의 서브픽셀로 구성되어 있다. 이를 독립적으로 제어하면 수평 해상도를 **약 3배**로 높일 수 있다.

<SubpixelComparison />

Microsoft의 **ClearType**(2000)과 FreeType의 **LCD 필터링**이 이 기술의 대표적 구현이다.

과정:

1. 글리프를 수평 해상도 3배로 래스터화 (각 픽셀당 3개 서브픽셀 샘플)
2. **LCD 필터(low-pass filter)**를 적용하여 컬러 프린지(color fringing)를 줄임
3. 결과를 RGB 채널에 분배

FreeType의 기본 LCD 필터 가중치는 `[0x08, 0x4D, 0x56, 0x4D, 0x08]` (정규화하면 약 `[0.03, 0.30, 0.34, 0.30, 0.03]`)이다. 이는 5-tap FIR 필터로, 인접 서브픽셀 간의 에너지 번짐을 줄인다.

```c filename="ftlcdfil.c (simplified)"
/* FreeType LCD 필터 적용 (수평 RGB 스트라이프) */
static void
_ft_lcd_filter_fir(FT_Bitmap* bitmap, FT_Byte* weights)
{
    /* weights = { 0x08, 0x4D, 0x56, 0x4D, 0x08 } */
    for each scanline row {
        for each pixel column {
            val = weights[0] * sample[col-2]
                + weights[1] * sample[col-1]
                + weights[2] * sample[col]
                + weights[3] * sample[col+1]
                + weights[4] * sample[col+2];
            output[col] = val >> 8;  /* 정규화 */
        }
    }
}
```

### 5.3 감마 보정

안티앨리어싱된 텍스트의 알파 블렌딩은 **선형 색공간**에서 수행해야 정확하다. 그러나 디스플레이는 비선형 감마 곡선을 적용한다 (일반적으로 <Math expr='gamma approx 2.2' />).

감마 보정 없이 블렌딩하면 중간 톤이 실제보다 어둡게 보여서, 얇은 획이 너무 가늘거나 텍스트가 전체적으로 가볍게 느껴진다.

올바른 파이프라인:

1. 커버리지 값 <Math expr='c' />를 구한다
2. 전경색/배경색을 선형 색공간으로 변환: <Math expr='L = S^gamma' />
3. 선형 공간에서 블렌딩: <Math expr='L_"out" = c dot L_"fg" + (1 - c) dot L_"bg"' />
4. 결과를 sRGB로 역변환: <Math expr='S_"out" = L_"out"^(1 slash gamma)' />

macOS의 Core Text는 이 보정을 내장하고 있다. Windows의 ClearType은 별도의 감마 테이블을 사용한다. FreeType 자체는 감마 보정을 수행하지 않으며, 이를 클라이언트 라이브러리에 위임한다.

### 5.4 고해상도 디스플레이의 영향

Retina/HiDPI 디스플레이(>= 200 PPI)에서는 서브픽셀 렌더링의 이점이 크게 줄어든다. macOS는 2018년(Mojave)부터 서브픽셀 AA를 기본 비활성화했다. 이는 다음 이유 때문이다:

- 200+ PPI에서 서브픽셀 단위의 해상도 향상은 시각적으로 무의미
- 서브픽셀 렌더링은 디스플레이 회전 시 RGB 스트라이프 방향이 바뀌면 아티팩트 발생
- 그레이스케일 AA가 고해상도에서 충분한 품질 제공
