# Vite 8은 어떻게 동작하는가

## 1. 개요 — Vite 8의 위치

Vite 8의 가장 큰 변화는 두 가지다:

1. **Rolldown 전환**: 번들러가 esbuild + Rollup 이중 체제에서 Rust 기반 Rolldown[^1]으로 통합되었다. dev 의존성 최적화와 production 빌드 모두 Rolldown을 사용한다.
2. **Environment API**: client/SSR/custom 런타임 환경을 독립적으로 처리하는 구조가 도입되었다. 각 환경이 자체 모듈 그래프, 플러그인 컨테이너, 의존성 최적화기를 가진다.

Vite 8의 핵심 코드는 `packages/vite/src/node/`에 있으며, 약 52,000줄의 TypeScript로 구성된다. `rolldown 1.0.0-rc.9`를 사용하며, esbuild는 일부 호환용(`transformWithEsbuild`)으로만 남아 있다.

## 3. 설정 시스템

`config.ts`의 `resolveConfig()`가 설정 해석의 진입점이다. 2,700줄에 달하는 이 함수는 인라인 설정, 설정 파일, 환경별 설정을 병합한다.

```typescript filename="packages/vite/src/node/config.ts"
// [focus:1-3]
export async function resolveConfig(
  inlineConfig: InlineConfig,
  command: 'build' | 'serve',
): Promise<ResolvedConfig> {
  // 1. 설정 파일 로드 (vite.config.ts)
  const loadResult = await loadConfigFromFile(/* ... */)

  // 2. 플러그인 정렬 (pre / normal / post)
  const [prePlugins, normalPlugins, postPlugins] = sortUserPlugins(rawUserPlugins)

  // 3. 환경별 설정 해석
  const environments: Record<string, ResolvedEnvironmentOptions> = {}
  for (const [name, envOptions] of Object.entries(config.environments)) {
    environments[name] = resolveEnvironmentOptions(envOptions, /* ... */)
  }

  // 4. 플러그인 파이프라인 구성
  ;(resolved.plugins as Plugin[]) = await resolvePlugins(
    resolved, prePlugins, normalPlugins, postPlugins,
  )
}
```

핵심은 `environments` 설정이다. Vite 8에서는 `client`와 `ssr` 외에 사용자 정의 환경을 추가할 수 있다. 각 환경은 독립적인 resolve 조건, 빌드 옵션, 최적화 설정을 가진다.

## 4. Environment API

Vite 8의 가장 근본적인 구조 변화다. 상속 체인은 다음과 같다:

```
PartialEnvironment → BaseEnvironment → DevEnvironment (dev)
                                     → BuildEnvironment (build)
                                     → ScanEnvironment (dep scan)
```

`PartialEnvironment`는 이름과 설정만 가진 최소 객체다. `config` 프로퍼티는 Proxy로 구현되어 환경별 설정과 최상위 설정을 투명하게 합친다:

```typescript filename="packages/vite/src/node/baseEnvironment.ts"
// [focus:1-8]
this.config = new Proxy(
  options as ResolvedConfig & ResolvedEnvironmentOptions,
  {
    get: (target, prop: keyof ResolvedConfig) => {
      if (prop in target) {
        return this._options[prop as keyof ResolvedEnvironmentOptions]
      }
      return this._topLevelConfig[prop]
    },
  },
)
```

`DevEnvironment`는 dev server에서 각 환경의 런타임을 담당한다. 핵심 구성:

```typescript filename="packages/vite/src/node/server/environment.ts"
export class DevEnvironment extends BaseEnvironment {
  mode = 'dev' as const
  moduleGraph: EnvironmentModuleGraph    // 환경별 독립 모듈 그래프
  depsOptimizer?: DepsOptimizer          // 환경별 독립 최적화기
  _pluginContainer: EnvironmentPluginContainer  // 환경별 플러그인 실행
}
```

서버 생성 시 모든 환경이 독립적으로 초기화된다:

```typescript filename="packages/vite/src/node/server/index.ts"
// [focus:1-7]
const environments: Record<string, DevEnvironment> = {}

await Promise.all(
  Object.entries(config.environments).map(
    async ([name, environmentOptions]) => {
      const environment = await environmentOptions.dev.createEnvironment(
        name, config, { ws },
      )
      environments[name] = environment
      await environment.init({ watcher, previousInstance })
    },
  ),
)
```

환경 유형별로 세 가지 구현체가 있다:

- **RunnableDevEnvironment**: 같은 Node.js 프로세스에서 모듈을 실행 (SSR 기본값)
- **FetchableDevEnvironment**: `Request`/`Response` 기반으로 모듈을 가져옴 (edge runtime 등)
- **FullBundleEnvironment**: 전체 번들을 메모리에 생성 (실험적 `bundledDev` 모드)

## 5. 플러그인 시스템

### 플러그인 순서

`plugins/index.ts`의 `resolvePlugins()`가 약 30개의 내장 플러그인을 정해진 순서로 배열한다:

```typescript filename="packages/vite/src/node/plugins/index.ts"
export async function resolvePlugins(
  config, prePlugins, normalPlugins, postPlugins,
): Promise<Plugin[]> {
  return [
    // 1. 의존성 최적화
    optimizedDepsPlugin(),
    // 2. 별칭 해석 (네이티브 Rolldown 또는 @rollup/plugin-alias)
    isBundled ? nativeAliasPlugin({ entries }) : aliasPlugin({ entries }),

    ...prePlugins,                    // 사용자 pre 플러그인

    // 3. 모듈 해석, HTML, CSS, JSON, 에셋 등
    ...oxcResolvePlugin(/* ... */),
    cssPlugin(config),
    oxcPlugin(config),                // OXC 변환 (esbuild 대체)
    nativeJsonPlugin(/* ... */),      // 네이티브 Rolldown JSON 플러그인
    assetPlugin(config),

    ...normalPlugins,                 // 사용자 normal 플러그인

    definePlugin(config),
    cssPostPlugin(config),

    ...postPlugins,                   // 사용자 post 플러그인
    ...buildPlugins.post,

    // 4. Dev 전용 (마지막)
    clientInjectionsPlugin(config),
    cssAnalysisPlugin(config),
    importAnalysisPlugin(config),     // import 재작성 (dev 핵심)
  ].filter(Boolean)
}
```

Vite 8에서 눈에 띄는 변화는 **네이티브 Rolldown 플러그인**의 도입이다. `nativeAliasPlugin`, `nativeJsonPlugin`, `nativeWasmFallbackPlugin` 등이 `rolldown/experimental`에서 제공되며, JS 플러그인보다 빠르게 동작한다.

### 플러그인 컨테이너

`pluginContainer.ts`는 원래 Preact의 wmr 프로젝트에서 가져온 코드로, Rollup 플러그인 API를 dev server 환경에서 실행한다. Vite 8에서는 환경별로 독립된 `EnvironmentPluginContainer`가 생성된다:

```typescript filename="packages/vite/src/node/server/pluginContainer.ts"
// Rollup 플러그인 훅을 순차적으로 실행
// resolveId → load → transform 체인
async function resolveId(rawId, importer, options) {
  for (const plugin of getSortedPlugins('resolveId')) {
    const handler = getHookHandler(plugin.resolveId)
    const result = await handler.call(ctx, rawId, importer, options)
    if (result) return result
  }
}
```

플러그인 훅 실행에는 **필터 최적화**가 적용된다. 플러그인이 `filter` 속성을 선언하면, ID나 코드 패턴에 매칭되지 않는 호출을 건너뛴다:

```typescript filename="packages/vite/src/node/plugins/index.ts"
// 캐시된 필터로 불필요한 플러그인 호출 스킵
export function getCachedFilterForPlugin(plugin, hookName) {
  let filters = filterForPlugin.get(plugin)
  // resolveId, load, transform 각각에 대해 필터 캐싱
}
```

## 6. Dev Server — 요청 처리 흐름

### 미들웨어 스택

`_createServer()`에서 connect 기반 미들웨어를 순서대로 등록한다:

```
1. timeMiddleware              — 디버그용 요청 타이머
2. rejectInvalidRequestMiddleware — 잘못된 요청 거부
3. rejectNoCorsRequestMiddleware  — CORS 없는 요청 거부
4. corsMiddleware              — CORS 헤더
5. hostValidationMiddleware    — DNS 리바인딩 방어
6. configureServer hooks       — 사용자 플러그인의 서버 미들웨어
7. cachedTransformMiddleware   — ETag 기반 304 캐시
8. proxyMiddleware             — 프록시 설정
9. baseMiddleware              — base 경로 처리
10. servePublicMiddleware      — /public 정적 파일
11. transformMiddleware        — ★ 핵심: 모듈 변환
12. serveStaticMiddleware      — 정적 파일
13. htmlFallbackMiddleware     — SPA 폴백
14. indexHtmlMiddleware        — index.html 변환
15. notFoundMiddleware         — 404
16. errorMiddleware            — 에러 처리
```

### transformMiddleware → transformRequest

브라우저가 `/src/App.tsx`를 요청하면 `transformMiddleware`가 이를 캐치한다:

```typescript filename="packages/vite/src/node/server/middlewares/transform.ts"
// JS/CSS 요청을 감지하여 transformRequest로 라우팅
if (isJSRequest(url) || isCSSRequest(url) || isImportRequest(url)) {
  const result = await environment.transformRequest(url)
  // ETag 설정 후 응답
  return send(req, res, result.code, type, { etag: result.etag, ... })
}
```

`transformRequest()`가 실제 변환을 수행한다:

```typescript filename="packages/vite/src/node/server/transformRequest.ts"
// [focus:1-3]
export function transformRequest(
  environment: DevEnvironment,
  url: string,
): Promise<TransformResult | null> {
  // 1. 캐시 확인 (모듈 그래프에서 transformResult)
  const cached = environment.moduleGraph.getModuleByUrl(url)
  if (cached?.transformResult) return cached.transformResult

  // 2. 플러그인 체인 실행
  //    resolveId → load → transform
  const id = await pluginContainer.resolveId(url)
  const loadResult = await pluginContainer.load(id)
  const transformResult = await pluginContainer.transform(code, id)

  // 3. 결과를 모듈 그래프에 캐시
  module.transformResult = { code, map, etag }
}
```

핵심은 **resolveId → load → transform** 파이프라인이다. 각 단계에서 등록된 모든 플러그인이 순서대로 실행되고, 처음으로 값을 반환한 플러그인의 결과가 사용된다.

### import 재작성

`importAnalysisPlugin`은 dev 모드의 핵심 플러그인이다. `es-module-lexer`로 import 구문을 파싱하고, bare import를 pre-bundled 경로로 재작성한다:

```typescript filename="packages/vite/src/node/plugins/importAnalysis.ts"
// import React from 'react'
// → import React from '/node_modules/.vite/deps/react.js?v=abc123'

// import './style.css'
// → import './style.css?t=1234567890'  (타임스탬프로 캐시 무효화)
```

이 플러그인이 `importAnalysis` 후 `postPlugins` 뒤에 배치되는 이유는, 다른 모든 플러그인의 변환이 끝난 최종 코드에서 import를 분석해야 하기 때문이다.

## 7. 모듈 그래프

`EnvironmentModuleNode`가 모듈 그래프의 노드다:

```typescript filename="packages/vite/src/node/server/moduleGraph.ts"
export class EnvironmentModuleNode {
  url: string                                    // 브라우저 요청 URL
  id: string | null                              // 파일시스템 경로 + 쿼리
  file: string | null                            // 파일시스템 경로
  type: 'js' | 'css' | 'asset'
  importers: Set<EnvironmentModuleNode>          // 이 모듈을 import하는 모듈들
  importedModules: Set<EnvironmentModuleNode>    // 이 모듈이 import하는 모듈들
  acceptedHmrDeps: Set<EnvironmentModuleNode>    // HMR accept 대상
  transformResult: TransformResult | null        // 캐시된 변환 결과
  lastHMRTimestamp: number
  invalidationState: TransformResult | 'HARD_INVALIDATED' | undefined
}
```

`importers`와 `importedModules`가 양방향 그래프를 형성한다. HMR에서 변경된 모듈의 영향 범위를 `importers`를 따라 올라가며 탐색한다.

`invalidationState`는 Vite 8의 **soft invalidation** 최적화를 위한 필드다. import 타임스탬프만 갱신하면 되는 경우(HMR 체인 내) 전체 재변환 없이 타임스탬프만 교체한다.

환경별 모듈 그래프 위에 `ModuleGraph` (mixedModuleGraph)가 하위 호환성을 위해 client/ssr 그래프를 통합한다:

```typescript filename="packages/vite/src/node/server/index.ts"
let moduleGraph = new ModuleGraph({
  client: () => environments.client.moduleGraph,
  ssr: () => environments.ssr.moduleGraph,
})
```

## 8. HMR (Hot Module Replacement)

### 서버 측: 변경 감지 → 업데이트 전송

파일 변경 시 chokidar 감시자가 이벤트를 발생시키고, `handleHMRUpdate()`가 처리한다:

```typescript filename="packages/vite/src/node/server/index.ts"
watcher.on('change', async (file) => {
  // 1. 모든 환경의 pluginContainer에 watchChange 알림
  await Promise.all(
    Object.values(server.environments).map((environment) =>
      environment.pluginContainer.watchChange(file, { event: 'update' }),
    ),
  )
  // 2. 모듈 그래프에서 해당 파일의 모듈 무효화
  for (const environment of Object.values(server.environments)) {
    environment.moduleGraph.onFileChange(file)
  }
  // 3. HMR 업데이트 처리
  await handleHMRUpdate('update', file, server)
})
```

`handleHMRUpdate()`는 변경된 파일에 대응하는 모듈을 찾고, `hotUpdate` 플러그인 훅을 실행한 뒤, 업데이트할 모듈 목록을 결정한다. 설정 파일이나 `.env`가 변경되면 서버를 재시작한다.

`updateModules()`가 실제 업데이트 경계를 탐색한다. `importers`를 따라 올라가면서 `import.meta.hot.accept()`를 선언한 모듈(HMR 경계)을 찾는다. 경계를 찾으면 해당 모듈까지의 업데이트 정보를 WebSocket으로 전송한다. 경계 없이 루트까지 도달하면 전체 페이지 리로드를 트리거한다.

### 클라이언트 측: 모듈 교체

`client/client.ts`가 브라우저에서 실행된다. WebSocket으로 서버의 HMR 페이로드를 수신한다:

```typescript filename="packages/vite/src/client/client.ts"
// WebSocket 연결
const transport = normalizeModuleRunnerTransport(
  createWebSocketModuleRunnerTransport({
    createConnection: () =>
      new WebSocket(
        `${socketProtocol}://${socketHost}?token=${wsToken}`,
        'vite-hmr',
      ),
    pingInterval: hmrTimeout,
  })
)

// HMR 핸들러가 update 페이로드를 처리
// → 변경된 모듈을 동적 import로 다시 로드
// → accept 콜백 실행
```

Vite 8에서는 `DevRuntime`(Rolldown의 실험적 기능)을 활용한 **번들 모드**도 지원한다. 이 모드에서는 개별 모듈 대신 번들을 메모리에 생성하여 제공한다.

## 9. 의존성 최적화 (Pre-bundling)

### 왜 필요한가

`node_modules`의 패키지는 CommonJS이거나, 수백 개의 내부 모듈로 분산되어 있다(lodash-es의 600+개 파일 등). 브라우저가 이를 개별 요청으로 로드하면 네트워크 폭포(waterfall)가 발생한다. Vite는 이를 사전에 단일 ESM 파일로 번들링한다.

### 스캔 → 번들링 → 런타임 발견

**1단계: 스캔** (`optimizer/scan.ts`)

`ScanEnvironment`를 생성하여 진입점에서 시작해 모든 bare import를 수집한다:

```typescript filename="packages/vite/src/node/optimizer/scan.ts"
// Rolldown의 scan API로 import 스캔
import { scan } from 'rolldown/experimental'

export class ScanEnvironment extends BaseEnvironment {
  mode = 'scan' as const
  // 최소한의 플러그인 컨테이너만 생성
}
```

**2단계: 번들링** (`optimizer/index.ts`)

스캔된 의존성을 Rolldown으로 번들링한다:

```typescript filename="packages/vite/src/node/optimizer/index.ts"
// [focus:1-3]
export async function runOptimizeDeps(
  environment: Environment,
  depsInfo: Record<string, OptimizedDepInfo>,
): Promise<DepOptimizationResult> {
  const rolldownOptions: RolldownOptions = {
    input: flatIdDeps,  // { 'react': '/abs/path/to/react/index.js', ... }
    plugins: [rolldownDepPlugin(environment, flatIdDeps, /* ... */)],
  }
  const result = await rolldown(rolldownOptions)
  await result.write(outputOptions)
}
```

결과물은 `node_modules/.vite/deps/`에 저장된다. 메타데이터 파일(`_metadata.json`)에 해시가 기록되어 캐시 유효성을 판단한다.

**3단계: 런타임 발견** (`optimizer/optimizer.ts`)

dev server 실행 중 새로운 의존성이 발견되면 `createDepsOptimizer`가 재번들링을 트리거한다:

```typescript filename="packages/vite/src/node/optimizer/optimizer.ts"
export function createDepsOptimizer(environment: DevEnvironment): DepsOptimizer {
  // 새 의존성 발견 시 100ms 디바운스 후 재번들링
  const debounceMs = 100

  function registerMissingImport(id: string, resolved: string) {
    // 새 의존성을 목록에 추가
    // 디바운스된 재번들링 트리거
    debouncedProcessing()
    // 번들링 완료까지 해당 모듈 요청을 대기시킴
    return optimizedDepInfo
  }
}
```

`importAnalysisPlugin`이 변환 중 최적화되지 않은 bare import를 발견하면 `registerMissingImport()`를 호출한다. 100ms 디바운스 후 발견된 모든 새 의존성을 한 번에 번들링하고, 페이지를 리로드한다.

## 10. 빌드

`build.ts`의 `build()` 함수가 프로덕션 빌드를 수행한다. Vite 8에서는 Rolldown의 `rolldown()` API를 직접 사용한다:

```typescript filename="packages/vite/src/node/build.ts"
// [focus:1-4]
import { rolldown } from 'rolldown'

export class BuildEnvironment extends BaseEnvironment {
  mode = 'build' as const
}

async function buildEnvironment(config, environment) {
  const bundle = await rolldown(rolldownOptions)
  await bundle.write(outputOptions)
}
```

`createBuilder`를 사용하면 여러 환경을 순차적으로 빌드할 수 있다. client 빌드의 결과(SSR manifest 등)를 SSR 빌드에서 참조하는 것이 가능하다.

빌드 시 dev 전용 플러그인(`importAnalysisPlugin`, `clientInjectionsPlugin` 등)은 제외되고, 대신 빌드 전용 플러그인(`buildImportAnalysisPlugin`, `manifestPlugin`, `ssrManifestPlugin` 등)이 추가된다.

## 11. 정리

Vite 8의 핵심 구조를 요약하면:

| 영역 | 핵심 파일 | 역할 |
|---|---|---|
| 설정 | `config.ts` | 설정 파일 로드, 환경별 머지, 플러그인 정렬 |
| 환경 | `baseEnvironment.ts`, `server/environment.ts` | 환경별 독립 모듈 그래프/플러그인/최적화기 |
| 플러그인 | `plugins/index.ts`, `server/pluginContainer.ts` | ~30개 내장 플러그인, Rollup 호환 실행기 |
| Dev 서버 | `server/index.ts` | connect 미들웨어 스택, 16단계 처리 |
| 변환 | `server/transformRequest.ts` | resolveId → load → transform 파이프라인 |
| 모듈 그래프 | `server/moduleGraph.ts` | import 관계 추적, HMR 경계 탐색 |
| HMR | `server/hmr.ts`, `client/client.ts` | 파일 변경 → 경계 탐색 → WebSocket → 모듈 교체 |
| 의존성 최적화 | `optimizer/` | 스캔 → Rolldown 번들링 → 런타임 발견 |
| 빌드 | `build.ts` | Rolldown 기반 프로덕션 번들링 |

esbuild + Rollup 이중 체제였던 Vite 5/6와 비교하면, Vite 8은 Rolldown으로 통합하면서 dev/build 간의 동작 차이를 줄였다. Environment API는 SSR을 "특수 케이스"가 아닌 "또 하나의 환경"으로 일반화하여, edge runtime이나 workerd 같은 다양한 실행 환경을 자연스럽게 지원할 수 있게 했다.
