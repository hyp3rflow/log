## 개요

Vite 8의 두 가지 핵심 변화는 **Rolldown 전환**과 **Environment API**다. esbuild와 Rollup의 이중 구조를 Rolldown 하나로 통합하고, client/SSR/custom 환경을 독립적으로 처리할 수 있는 Environment API를 도입했다. 이 글에서는 `packages/vite/src/node/` 디렉토리의 약 39,000줄의 소스코드를 직접 읽고, dev server의 요청 처리부터 프로덕션 빌드까지의 내부 구조를 분석한다.

## 설정 시스템

`config.ts`의 `resolveConfig()`가 전체 설정을 해석한다. 약 2,700줄의 이 함수는 인라인 설정, 설정 파일, 기본값을 병합한다.


## Environment API

Vite 8의 가장 중요한 아키텍처 변경이다. 상속 체인은 다음과 같다:

```text filename="상속 체인"
PartialEnvironment
  └── BaseEnvironment
        ├── DevEnvironment    (mode = 'dev')
        ├── BuildEnvironment  (mode = 'build')
        └── ScanEnvironment   (mode = 'scan')
```

### PartialEnvironment

가장 기본이 되는 클래스다. 환경 이름, 설정 프록시, 환경별 로거를 제공한다.

```ts filename="baseEnvironment.ts"
export class PartialEnvironment {
  name: string
  config: ResolvedConfig & ResolvedEnvironmentOptions
  logger: Logger

  constructor(
    name: string,
    topLevelConfig: ResolvedConfig,
    options: ResolvedEnvironmentOptions = topLevelConfig.environments[name],
  ) {
    this.name = name
    this._topLevelConfig = topLevelConfig
    this._options = options
    this.config = new Proxy(
      options as ResolvedConfig & ResolvedEnvironmentOptions,
      {
        get: (target, prop: keyof ResolvedConfig) => {
          if (prop in target) {
            return this._options[prop as keyof ResolvedEnvironmentOptions]
          }
          return this._topLevelConfig[prop]
        },
      },
    )
    // ...
  }
}
```

`config` 프로퍼티는 Proxy로 구현된다. 환경별 옵션이 있으면 그것을, 없으면 최상위 설정에서 값을 가져온다. 이 설계 덕분에 환경별 설정 오버라이드가 자연스럽게 동작한다.

### DevEnvironment

dev 서버에서 사용되는 환경 클래스다. 환경별로 **독립적인 모듈 그래프**와 **플러그인 컨테이너**를 가진다.

### perEnvironmentState

플러그인이 환경별 상태를 관리할 수 있도록 `perEnvironmentState` 유틸리티를 제공한다:

```ts filename="environment.ts"
export function perEnvironmentState<State>(
  initial: (environment: Environment) => State,
): (context: PluginContext) => State {
  const stateMap = new WeakMap<Environment, State>()
  return function (context: PluginContext) {
    const { environment } = context
    let state = stateMap.get(environment)
    if (!state) {
      state = initial(environment)
      stateMap.set(environment, state)
    }
    return state
  }
}
```

WeakMap을 사용해 환경 인스턴스별로 상태를 분리한다. 플러그인이 client와 SSR 환경에서 서로 다른 상태를 유지해야 할 때 사용한다.

## 플러그인 시스템

### resolvePlugins()

`plugins/index.ts`의 `resolvePlugins()`가 전체 플러그인 배열을 조립한다. 순서가 중요하다:

```ts filename="plugins/index.ts"
export async function resolvePlugins(
  config: ResolvedConfig,
  prePlugins: Plugin[],
  normalPlugins: Plugin[],
  postPlugins: Plugin[],
): Promise<Plugin[]> {
  const isBuild = config.command === 'build'
  const isBundled = config.isBundled

  return [
    // 1. 의존성 최적화 (dev only)
    !isBundled ? optimizedDepsPlugin() : null,
    // 2. 별칭 해석 — 네이티브 또는 @rollup/plugin-alias
    isBundled && !config.resolve.alias.some((v) => v.customResolver)
      ? nativeAliasPlugin({ entries: config.resolve.alias.map(/*...*/) })
      : aliasPlugin({ entries: config.resolve.alias }),

    ...prePlugins,

    // 3. OXC 기반 모듈 해석
    ...oxcResolvePlugin({ root: config.root, /* ... */ }),
    // 4. CSS, OXC 트랜스파일, JSON, WASM, asset 등
    cssPlugin(config),
    config.oxc !== false ? oxcPlugin(config) : null,
    nativeJsonPlugin({ ...config.json, minify: isBuild }),
    wasmHelperPlugin(),
    assetPlugin(config),

    ...normalPlugins,

    // 5. define 치환, CSS 후처리, 빌드 플러그인
    definePlugin(config),
    cssPostPlugin(config),
    ...buildPlugins.pre,

    ...postPlugins,

    ...buildPlugins.post,

    // 6. dev 전용: 클라이언트 주입, CSS 분석, import 분석
    ...(isBundled ? [] : [
      clientInjectionsPlugin(config),
      cssAnalysisPlugin(config),
      importAnalysisPlugin(config),
    ]),
  ].filter(Boolean) as Plugin[]
}
```

주목할 점:

- **네이티브 플러그인 도입**: `nativeAliasPlugin`, `nativeJsonPlugin`, `nativeWasmFallbackPlugin`은 `rolldown/experimental`에서 가져온다. Rust로 구현된 이 플러그인들은 JS 플러그인보다 빠르다.
- **dev/build 분기**: `isBundled` 플래그로 dev 전용 플러그인과 build 전용 플러그인을 구분한다.
- **importAnalysisPlugin은 항상 마지막**: dev 모드에서 모든 변환이 끝난 후 import를 재작성해야 하므로 배열의 가장 끝에 위치한다.

### 플러그인 훅 정렬

각 훅 실행 시 `order` 필드에 따라 `pre → normal → post` 순서로 정렬된다:

```ts filename="plugins/index.ts"
export function getSortedPluginsByHook<K extends keyof Plugin>(
  hookName: K,
  plugins: readonly Plugin[],
): PluginWithRequiredHook<K>[] {
  const sortedPlugins: Plugin[] = []
  let pre = 0, normal = 0, post = 0
  for (const plugin of plugins) {
    const hook = plugin[hookName]
    if (hook) {
      if (typeof hook === 'object') {
        if (hook.order === 'pre') {
          sortedPlugins.splice(pre++, 0, plugin)
          continue
        }
        if (hook.order === 'post') {
          sortedPlugins.splice(pre + normal + post++, 0, plugin)
          continue
        }
      }
      sortedPlugins.splice(pre + normal++, 0, plugin)
    }
  }
  return sortedPlugins as PluginWithRequiredHook<K>[]
}
```

### 요청 처리 흐름

브라우저가 `/src/App.tsx`를 요청하면:

`transformMiddleware`가 요청을 가로채 `transformRequest()`를 호출한다:

```ts filename="server/middlewares/transform.ts"
return async function viteTransformMiddleware(req, res, next) {
  const environment = server.environments.client
  // ...
  if (isJSRequest(url) || isImportRequest(url) || isCSSRequest(url)) {

```

`transformRequest()`는 `doTransform()`을 호출하며, 동일 URL에 대한 중복 요청을 `_pendingRequests` Map으로 디바운싱한다:

```ts filename="server/transformRequest.ts"
export function transformRequest(
  environment: DevEnvironment,
  url: string,
  options: TransformOptionsInternal = {},
): Promise<TransformResult | null> {
  const timestamp = monotonicDateNow()
  url = removeTimestampQuery(url)

  const pending = environment._pendingRequests.get(url)
  if (pending) {
    return environment.moduleGraph.getModuleByUrl(url).then((module) => {
      if (!module || pending.timestamp > module.lastInvalidationTimestamp) {
        return pending.request
      } else {
        pending.abort()
        return transformRequest(environment, url, options)
      }
    })
  }

  const request = doTransform(environment, url, options, timestamp)
  environment._pendingRequests.set(url, { request, timestamp, abort: clearCache })
  return request.finally(clearCache)
}
```

`doTransform()` 내부의 `loadAndTransform()`에서 실제 플러그인 체인이 실행된다:

```ts filename="server/transformRequest.ts"
async function loadAndTransform(
  environment: DevEnvironment,
  id: string, url: string, options, timestamp, mod?, resolved?,
) {
  const { pluginContainer } = environment

  // 1. load 훅 실행
  const loadResult = await pluginContainer.load(id)
  if (loadResult == null) {
    // fallback: 파일 시스템에서 직접 읽기
    code = await fsp.readFile(cleanUrl(id), 'utf-8')
  }

  // 2. 모듈 그래프에 등록
  const mod = await moduleGraph._ensureEntryFromUrl(url, undefined, resolved)

  // 3. transform 훅 체인 실행
  const transformResult = await pluginContainer.transform(code, id, {
    inMap: map,
  })

  // 4. 결과를 모듈 그래프에 캐싱
  mod.transformResult = { code, map, etag }
  return mod.transformResult
}
```

## 모듈 그래프

### EnvironmentModuleNode

각 모듈은 `EnvironmentModuleNode`로 표현된다:

```ts filename="server/moduleGraph.ts"
export class EnvironmentModuleNode {
  environment: string
  url: string                                    // 공개 URL (/src/App.tsx)
  id: string | null = null                       // 절대 파일 경로
  file: string | null = null
  type: 'js' | 'css' | 'asset'
  importers: Set<EnvironmentModuleNode> = new Set()
  importedModules: Set<EnvironmentModuleNode> = new Set()
  acceptedHmrDeps: Set<EnvironmentModuleNode> = new Set()
  isSelfAccepting?: boolean
  transformResult: TransformResult | null = null
  lastInvalidationTimestamp = 0
  invalidationState: TransformResult | 'HARD_INVALIDATED' | undefined
}
```

핵심 필드:
- `importers`: 이 모듈을 import하는 모듈들 (역방향 참조)
- `importedModules`: 이 모듈이 import하는 모듈들 (정방향 참조)
- `acceptedHmrDeps`: `import.meta.hot.accept()`로 등록된 HMR 수용 모듈
- `transformResult`: 변환된 코드의 캐시

### EnvironmentModuleGraph

환경별 모듈 그래프는 4개의 인덱스를 유지한다:

```ts filename="server/moduleGraph.ts"
export class EnvironmentModuleGraph {
  environment: string
  urlToModuleMap: Map<string, EnvironmentModuleNode> = new Map()
  idToModuleMap: Map<string, EnvironmentModuleNode> = new Map()
  etagToModuleMap: Map<string, EnvironmentModuleNode> = new Map()
  fileToModulesMap: Map<string, Set<EnvironmentModuleNode>> = new Map()
}
```

URL, ID(절대 경로), ETag, 파일 경로 — 네 가지 방법으로 모듈을 조회할 수 있다. `etagToModuleMap`은 `cachedTransformMiddleware`에서 304 응답을 빠르게 확인하는 데 사용된다.

서버 레벨에서는 `ModuleGraph` (mixedModuleGraph)가 환경별 그래프를 통합하여 하위 호환성을 제공한다.

## HMR

### 파일 변경 감지

chokidar가 파일 변경을 감지하면 `handleHMRUpdate()`가 호출된다:


### handleHMRUpdate

설정 파일이나 `.env` 파일이 변경되면 서버를 재시작한다. 그 외의 경우 각 환경에서 영향받는 모듈을 찾는다:

```ts filename="server/hmr.ts"
export async function handleHMRUpdate(
  type: 'create' | 'delete' | 'update',
  file: string,
  server: ViteDevServer,
): Promise<void> {
  const isConfig = file === config.configFile
  if (isConfig || isConfigDependency || isEnv) {
    await restartServerWithUrls(server)
    return
  }

  for (const environment of Object.values(server.environments)) {
    const mods = new Set(environment.moduleGraph.getModulesByFile(file))
    const options = {
      ...contextMeta,
      modules: [...mods],
    }
    hotMap.set(environment, { options })
  }
  // ...
}
```

### updateModules

영향받는 모듈에서 HMR 경계를 찾기 위해 `propagateUpdate()`로 모듈 그래프를 역방향 탐색한다:

```ts filename="server/hmr.ts"
export function updateModules(
  environment: DevEnvironment,
  file: string,
  modules: EnvironmentModuleNode[],
  timestamp: number,
): void {
  const updates: Update[] = []
  let needFullReload = modules.length === 0

  for (const mod of modules) {
    const boundaries: PropagationBoundary[] = []
    const hasDeadEnd = propagateUpdate(mod, traversedModules, boundaries)

    environment.moduleGraph.invalidateModule(mod, invalidatedModules, timestamp, true)

    if (hasDeadEnd) {
      needFullReload = hasDeadEnd
      continue
    }

    updates.push(
      ...boundaries.map(({ boundary, acceptedVia }) => ({
        type: `${boundary.type}-update` as const,
        timestamp,
        path: normalizeHmrUrl(boundary.url),
        acceptedPath: normalizeHmrUrl(acceptedVia.url),
      })),
    )
  }

  if (needFullReload) {
    hot.send({ type: 'full-reload', triggeredBy: path.resolve(config.root, file) })
    return
  }

  hot.send({ type: 'update', updates })
}
```

변경된 모듈에서 위쪽으로 올라가면서 `import.meta.hot.accept()`를 선언한 모듈(HMR boundary)을 찾는다. boundary를 찾으면 해당 모듈만 교체(partial update)하고, 찾지 못하면 전체 페이지를 새로고침(full reload)한다. 업데이트 메시지는 WebSocket을 통해 클라이언트로 전송된다.

## 의존성 최적화

`node_modules`의 의존성은 CommonJS일 수 있고, 수백 개의 내부 모듈을 가질 수 있다. Vite는 이를 사전 번들링(pre-bundling)하여 ESM으로 변환하고 단일 파일로 묶는다.

### 스캔 단계

`optimizer/scan.ts`의 `ScanEnvironment`가 프로젝트의 import를 스캔한다:

```ts filename="optimizer/scan.ts"
export class ScanEnvironment extends BaseEnvironment {
  mode = 'scan' as const

  get pluginContainer(): EnvironmentPluginContainer {
    if (!this._pluginContainer)
      throw new Error(
        `${this.name} environment.pluginContainer called before initialized`,
      )
    return this._pluginContainer
  }

  async init(): Promise<void> {
    if (this._initiated) return
    this._initiated = true
    this._pluginContainer = await createEnvironmentPluginContainer(
      this, this.plugins, undefined, false,
    )
  }
}
```

Rolldown의 `scan()` API(`rolldown/experimental`)를 사용해 엔트리포인트에서 시작하여 bare import(예: `import React from 'react'`)를 수집한다.

### Pre-bundling

`optimizer/index.ts`의 `runOptimizeDeps()`가 수집된 의존성을 Rolldown으로 번들링한다:

```ts filename="optimizer/index.ts"
export function runOptimizeDeps(
  environment: Environment,
  depsInfo: Record<string, OptimizedDepInfo>,
): {
  cancel: () => Promise<void>
  result: Promise<DepOptimizationResult>
} {
  const depsCacheDir = getDepsCacheDir(environment)
  const processingCacheDir = getProcessingDepsCacheDir(environment)

  // 임시 디렉토리에서 작업 후 원자적으로 교체
  fs.mkdirSync(processingCacheDir, { recursive: true })
  fs.writeFileSync(
    path.resolve(processingCacheDir, 'package.json'),
    `{\n  "type": "module"\n}\n`,
  )
  // ...
}
```

번들링은 임시 디렉토리에서 수행되며, 완료 후 rename으로 원자적으로 캐시 디렉토리를 교체한다. 이는 프로세스가 중단되어도 캐시가 손상되지 않도록 보장한다.

결과는 `node_modules/.vite/deps/`에 저장되며, 각 의존성은 `flattenId(dep.id) + '.js'` 형식의 파일명을 가진다.

### 런타임 발견

`optimizer/optimizer.ts`의 `createDepsOptimizer()`는 dev 서버 실행 중 새로운 의존성이 발견되면 재번들링을 트리거한다:

```ts filename="optimizer/optimizer.ts"
export function createDepsOptimizer(
  environment: DevEnvironment,
): DepsOptimizer {
  const sessionTimestamp = Date.now().toString()
  let debounceProcessingHandle: NodeJS.Timeout | undefined

  const { noDiscovery, holdUntilCrawlEnd } = options

  const depsOptimizer: DepsOptimizer = {
    init,
    metadata: initDepsOptimizerMetadata(environment, sessionTimestamp),
    registerMissingImport: (id, resolved) => {
      // 새 의존성 발견 시 debounce 후 재번들링
    },
    // ...
  }
  return depsOptimizer
}
```

`registerMissingImport()`가 호출되면 100ms debounce 후 `runOptimizeDeps()`를 재실행한다. 재번들링 중에는 해당 의존성의 요청을 promise로 보류하고, 완료되면 resolve하여 응답한다.

## 빌드

### BuildEnvironment

프로덕션 빌드 시 `BuildEnvironment`가 사용된다:

```ts filename="build.ts"
export class BuildEnvironment extends BaseEnvironment {
  mode = 'build' as const
  // ...
}
```

### Rolldown 기반 번들링

`build.ts`의 `resolveRolldownOptions()`가 Rolldown 설정을 구성한다:

```ts filename="build.ts"
export function resolveRolldownOptions(
  environment: Environment,
  chunkMetadataMap: ChunkMetadataMap,
): RolldownOptions {
  const plugins = environment.plugins.map((p) =>
    injectEnvironmentToHooks(environment, chunkMetadataMap, p),
  )

  const rollupOptions: RolldownOptions = {
    input,
    plugins,
    external: options.rollupOptions.external,
    transform: {
      target: options.target === false ? undefined : options.target,
    },
    experimental: {
      viteMode: true,
    },
  }
  return rollupOptions
}
```

`experimental.viteMode: true`는 Rolldown에게 Vite 모드로 동작하도록 지시하는 플래그다.

`build()` 함수는 `createBuilder()`를 통해 환경별 독립 빌드를 지원한다:

```ts filename="build.ts"
export async function build(
  inlineConfig: InlineConfig = {},
): Promise<RolldownOutput | RolldownOutput[] | RolldownWatcher> {
  const builder = await createBuilder(inlineConfig, true)
  const environment = Object.values(builder.environments)[0]
  if (!environment) throw new Error('No environment found')
  return builder.build(environment)
}
```

### 빌드 플러그인 구성

빌드 시에는 dev에는 없는 추가 플러그인이 투입된다:

```ts filename="build.ts"
export async function resolveBuildPlugins(config: ResolvedConfig): Promise<{
  pre: Plugin[]
  post: Plugin[]
}> {
  return {
    pre: [
      prepareOutDirPlugin(),
      perEnvironmentPlugin('vite:rollup-options-plugins', async (environment) =>
        (await asyncFlatten(
          arraify(environment.config.build.rollupOptions.plugins),
        )).filter(Boolean) as Plugin[],
      ),
    ],
    post: [
      ...buildImportAnalysisPlugin(config),
      ...(config.build.minify === 'esbuild' ? [buildEsbuildPlugin()] : []),
      ...terserPlugin(config),
      licensePlugin(),
      manifestPlugin(),
      ssrManifestPlugin(),
      buildReporterPlugin(config),
      nativeLoadFallbackPlugin(),
    ],
  }
}
```

## 정리

Vite 8의 내부 구조를 요약하면:

1. **Rolldown 통합**: dev의 의존성 최적화와 production 빌드 모두 Rolldown을 사용한다. esbuild는 호환용(minification 등)으로만 남아있다.
2. **Environment API**: `PartialEnvironment → BaseEnvironment → DevEnvironment/BuildEnvironment` 상속 체인으로, 환경별로 독립적인 모듈 그래프, 플러그인 컨테이너, 의존성 최적화기를 가진다.
3. **플러그인 파이프라인**: Rollup 호환 컨테이너 위에 네이티브 Rolldown 플러그인을 도입하여 성능을 개선했다. `resolvePlugins()`가 약 30개의 내장 플러그인을 정확한 순서로 조립한다.
4. **Dev Server**: connect 미들웨어 스택 → `transformMiddleware` → `transformRequest()` → resolve/load/transform 플러그인 체인 → ESM 응답이라는 파이프라인으로 동작한다.
5. **HMR**: 파일 변경 → 모듈 그래프 역방향 탐색 → HMR boundary 발견 → WebSocket으로 클라이언트에 업데이트 전송.
6. **의존성 최적화**: 스캔 → Rolldown pre-bundle → 임시 디렉토리에서 원자적 교체. 런타임에 새 의존성 발견 시 debounce 후 재번들링.

전체 약 39,000줄의 코드가 이 구조 위에서 동작한다.
